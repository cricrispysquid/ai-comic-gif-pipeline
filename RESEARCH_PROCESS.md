# AI / Agent 生成漫画风技术 GIF：调研与迭代过程

## 1. 提出的问题

LinkedIn 技术文章中使用的精美 GIF，是否可以由 AI / Agent 高效产生？如果输入是一篇音频 AI、机器学习或 Agent 论文，能否让系统理解文章内容，并自动生成类似的漫画风解释动画？

参考帖子：

- [Schema-Guided Agent Memory — LinkedIn](https://www.linkedin.com/posts/avi-chawla_i-just-fixed-my-agent-memory-using-pydantic-share-7479888244239826944-vgTv/)

这个问题包含两个不同层面：

1. AI 能不能理解文章并设计解释顺序？
2. AI 能不能稳定地产生精美、准确、风格一致的动画？

我们在实验中逐步发现，第二个问题不能仅靠一次文生图或文生视频调用解决，而需要一个由多个阶段组成的 Agent 流水线。

## 2. 第一阶段：验证程序化动画是否可行

最初以 cross-attention masking 为主题做了一个 Canvas + GIFEncoder 原型。动画包含 decoder query、encoder memory、PAD token、mask 和注意力连线，验证了以下链路：

![第一阶段的 Cross-Attention Masking Canvas + GIFEncoder 原型](research_assets/cross_attention_prototype.gif)



```text
自然语言概念
    ↓
Agent 设计解释步骤
    ↓
Agent 生成 Canvas 动画代码
    ↓
逐帧渲染
    ↓
GIF
```

这个实验说明：AI 不一定要直接生成每个像素。AI 也可以理解概念、设计分镜、编写动画程序，再运行和检查结果。这种方式对公式、文字、节点和箭头更可靠，也容易复现和修改。

但第一版采用深色背景、发光节点和规整几何图形，视觉上是典型的“简约理工风”，与老师所指的例子存在明显差距。

## 3. 重要转折：程序正确不等于视觉风格正确

拿到具体 LinkedIn 链接后，我们下载并检查了公开 GIF：

- 尺寸：1200 × 1195
- 帧数：41 帧
- 帧间隔：50 ms
- 完整循环：约 2.05 秒
- 文件大小：约 1.18 MB

原作的视觉特征包括：

- 白纸背景和大量留白；
- 粉彩蓝、绿、紫色圆角容器；
- 类手写标题和正文；
- 粗细略有变化的手绘轮廓；
- 卡通人物、AI 脑、聊天气泡、文件、放大镜、链条、时钟和图谱等图标；
- 严格准确的文字、箭头、步骤编号和流程关系；
- 角色轻微浮动、箭头流动、卡片强调等短循环动画。

41 帧和约两秒的时长也说明：它不是复杂的镜头动画，而更接近“完成度很高的静态信息图 + 少量局部动态”。精美感主要来自插画、版式、字体和色彩，而不是动画幅度。

这一步修正了最初的技术判断：仅靠程序绘制几何图形，可以得到准确动画，但无法自动获得参考作品的插画质感。

## 4. 为什么不直接使用文生视频

我们比较了直接文生视频与程序化动画的特点。

直接生成视频的优势是快速、画面丰富，适合人物场景、氛围镜头和没有精确文字的内容。但用于论文信息图时容易出现：

- 文字拼写错误；
- 不同帧中的角色和图标发生变化；
- 箭头方向和节点关系不稳定；
- 公式或专业术语被改写；
- 首尾不能形成无缝循环；
- 修改一个局部时整幅画面发生漂移。

因此，直接文生视频可以作为辅助工具，但不适合作为科学信息图的唯一渲染器。

## 5. 得到的核心分工

最终形成了一个简洁的原则：

> 图像模型负责“可爱”，Agent 和程序负责“正确”。

具体来说：

- LLM / Content Agent：理解论文，提取主旨、术语、节点和解释顺序；
- Art Director Agent：决定漫画比喻、版式、色板、角色和图标需求；
- 图像模型：生成卡通人物和不要求文字精度的插画资产；
- SVG / Canvas / Remotion：准确放置文字、卡片、箭头和步骤；
- Critic Agent：检查关键帧，发现溢出、遮挡、风格漂移和叙事问题；
- Renderer：输出 MP4、WebM 或 GIF。

## 6. 技术路线比较

### Remotion + SVG

最适合最终的规模化版本。React 组件可以建立可复用模板，SVG 负责清晰的文字和矢量图形，Remotion 负责时间轴、预览和批量渲染。

- [Remotion 官方网站](https://www.remotion.dev/)

### Rive

适合制作角色状态机，例如正常、眨眼、思考、开心等状态，也适合交互式动画。但自动生成复杂信息图版式不是它最强的部分。

- [Rive State Machines](https://rive.app/docs/editor/state-machine/state-machine)

### Lottie

适合轻量矢量循环动画和网页/App 播放。优点是文件小、跨平台；缺点是自动生成和调试复杂 Lottie JSON 的成本较高。

- [Lottie 格式文档](https://lottiefiles.github.io/lottie-docs/Introduction/)

### Manim

适合数学公式变形、几何对象和教学式推导。对于漫画人物、手绘图标和复杂信息图排版，需要另外建立视觉资产系统。

- [Manim Community 文档](https://docs.manim.community/en/stable/tutorials/building_blocks.html)

### Canvas + GIFEncoder

适合快速验证，也是当前 MVP 使用的方案。依赖少、渲染确定、容易由 Agent 修改，但复杂排版和专业插画能力不如 SVG + Remotion。

## 7. 长文章带来的新问题

当输入从一个短概念变成一篇长文章时，不能继续尝试把所有信息塞进一张图。常见失败包括：

- 字体被压得过小；
- 每个段落都被当成同等重要；
- 多张图重复相同背景介绍；
- 角色、配色和术语在不同页面发生变化；
- 页面之间没有完整的叙事顺序。

我们讨论过“让 AI 按段落逐张生成图片”，但最终认为这仍然不够。段落只是文章的排版单位，不一定是最佳的视觉叙事单位。一个长段落可能需要拆成多个步骤，多个短段也可能应该合并成一页。

因此采用两阶段策略。

### 第一遍：全文规划

AI 阅读全文，生成：

- 一句话主旨；
- 全局章节和阅读顺序；
- 统一术语表；
- 内容取舍；
- 固定角色、色板和插画风格；
- 每章预计需要的页面数量。

### 第二遍：逐章节分镜

在全局约束下处理每个章节，将内容整理成每页 1–3 个信息卡。AI 只生成结构化 storyboard，不直接生成整张图。

```text
全文
  ↓
全局 outline + style bible + glossary
  ↓
章节 1 storyboard
章节 2 storyboard
章节 3 storyboard
  ↓
统一模板渲染
```

这样既能分批处理长文，也能维持跨页面一致性。

## 8. 数据契约的重要性

为了让系统稳定，我们不让 LLM 每次自由编写完整动画程序，而是让它输出受限制的 `storyboard.json`：

```json
{
  "title": "文章标题",
  "style": "pastel-handdrawn",
  "pages": [
    {
      "section": "章节标题",
      "cards": [
        {
          "title": "要点标题",
          "body": "简短解释",
          "icon": "brain"
        }
      ]
    }
  ]
}
```

结构化输出带来的好处是：

- 可以在渲染前检查缺失字段；
- 可以限制每页信息量；
- 可以验证箭头端点和步骤编号；
- 人可以直接修改 JSON，不必修改动画代码；
- 相同内容可以切换不同模板；
- 生成失败时只修复某一页或某一个字段。

## 9. MVP 实现与实际迭代

我们最终实现了 `comic_pipeline.js`。程序能够读取 Markdown、纯文本或 storyboard JSON，并输出多张漫画风 GIF。

第一次长文章测试中，每页最多放四个卡片。程序成功运行，但关键帧检查发现两个问题：

1. 英文按照字符换行，导致单词被截断；
2. 四卡页面过密，正文接近卡片底部，阅读体验较差。

随后进行了局部修复：

- 英文改为按单词换行，中文保持按字符换行；
- 每页最大卡片数量由四个降为三个；
- 长章节自动拆成连续页面；
- 保留页面编号和 manifest，维持整体阅读顺序。

重新渲染后，示例长文章被转换成五张连续 GIF。这一过程也证明了“渲染后反馈 + 局部修复”是必要的。仅仅确认代码执行成功，并不能保证信息图可读。

近期研究也在探索类似的方向：通过显式视觉规划、共享场景状态、渲染反馈和局部修复，提高自动教育动画的可靠性。

- [OmniManim: Render-feedback-aware Educational Animation Generation](https://arxiv.org/abs/2605.15585)
- [Preacher: Paper-to-Video Agentic System](https://www.openaccess.thecvf.com/content/ICCV2025/papers/Liu_Preacher_Paper-to-Video_Agentic_System_ICCV_2025_paper.pdf)

## 10. 效率来自哪里

提高效率的关键并不是更强的文生视频，而是：

1. 固定数据格式
LLM 只输出受约束的 `storyboard.json`，不允许每篇文章自由生成整套动画代码。

2. 建立资产库
预先准备：
- 2–3 个漫画研究员角色
- 30–50 个常用 SVG 图标
- 4 个信息图模板
- 2–3 套粉彩色板
- 6 个局部循环动画预设

3. 只做局部小动画
例如角色浮动、眨眼、箭头流动、卡片呼吸、图标弹出。这正是参考 GIF 用 41 帧、2.05 秒达到精致效果的方式。

4. 渲染后反馈
自动抽取关键帧，检查文字溢出、遮挡、错误连线、风格漂移和循环断点，只修出错的局部。近期教育动画研究也正在采用“显式视觉规划、渲染反馈和局部修复”的架构，而不是一次性生成全部画面
参考：https://arxiv.org/abs/2605.15585


工程上的大致效率：
- 首个模板和资产系统：2–5 个工作日
- 同一模板换一篇新文章：5–20 分钟
- 需要少量新插画：20–45 分钟
- 新版式：半天到一天，加入模板库后可持续复用

## 11. 当前结论

AI / Agent 可以高效地产生类似的论文解释 GIF，但可靠方案不是“一句话直接生成最终 GIF”，而是：

```text
理解全文
→ 规划内容
→ 结构化分镜
→ 复用漫画资产
→ 程序化排版与局部动画
→ 渲染后检查
→ 局部修复
```

当前调研已经验证了文章分段、结构化分镜、模板渲染、多页 GIF 和迭代检查，并据此确定最终实现应采用“两阶段 LLM + 结构化数据契约 + 本地确定性渲染”。
